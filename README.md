# Booking Service

REST API для создания и обработки записей на встречи. После создания брони задача отправляется в очередь Celery, где подтверждается либо завершается ошибкой с имитацией сбоя внешнего сервиса.

## Стек технологий

* FastAPI
* SQLAlchemy
* PostgreSQL
* Alembic
* Celery
* Redis
* Docker Compose
* Pytest

## Запуск проекта

Клонировать репозиторий:
```bash
git clone <repository_url>
cd <repository_name>
```

Создать файл окружения:
```bash
cp .env.example .env
```

Запустить проект:
```bash
docker-compose up --build
```

После запуска будут доступны:

* FastAPI API
* PostgreSQL
* Redis
* Celery Worker

## Запуск тестов

Для запуска тестов зайти в контейнер `fastapi-app` через bash (нарушает ТЗ, но так действительно проще и быстрее):
```bash
/test.sh
```

Тестами покрыты:
* создание брони;
* получение существующей и несуществующей брони;
* получение списка с фильтрацией и пагинацией;
* удаление брони в допустимом и недопустимом статусах;
* логика Celery-воркера с использованием mock внешнего вызова.

## Принятые технические решения

### FastAPI

В качестве основного фреймворка выбран FastAPI благодаря поддержке асинхронной обработки запросов, удобной работе с Pydantic и высокой производительности.

### SQLAlchemy + Alembic

Для работы с базой данных используется SQLAlchemy ORM, а управление схемой базы выполняется через Alembic. Такой подход позволяет отказаться от ручных SQL-запросов и обеспечивает удобное управление миграциями.

### Celery + Redis

После создания брони API не выполняет её подтверждение самостоятельно, а ставит задачу в очередь Celery. Redis используется одновременно как broker сообщений и backend результатов. Это позволяет отделить обработку бизнес-логики от HTTP-запросов и не блокировать клиента во время выполнения длительных операций.

### Идемпотентность задачи

Для обеспечения идемпотентности задача `confirm_booking` в начале выполнения проверяет текущий статус брони. Если статус уже является финальным (`CONFIRMED` или `FAILED`), выполнение немедленно завершается с соответствующим результатом, минуя бизнес-логику и имитацию сбоев.

Такой подход гарантирует, что повторные вызовы с тем же `booking_id` не изменят состояние системы и не вызовут побочных эффектов (например, повторной отправки уведомлений или изменения статуса).

### Имитация внешнего сервиса

В процессе обработки задачи с вероятностью 15% имитируется ошибка внешнего сервиса. В этом случае статус брони изменяется на `FAILED`. При успешном выполнении бронь получает статус `CONFIRMED`, а в лог записывается mock-уведомление.

### Структура проекта

Проект разделён на независимые компоненты:

* API — обработка HTTP-запросов;
* слой работы с базой данных;
* Celery Worker — фоновая обработка задач;
* миграции Alembic;
* тесты API и воркера.
